# Cross-Squad Workflows

Workflows que atravessam 2+ squads são **first-class** (não improvisados). Vivem aqui em `primeteam-ops/workflows/cross-squad/` porque @ops-chief é o **master orchestrator** que coordena cross-squad handoffs.

## Filosofia

Antes desta refactor (2026-05-14), squads operavam paralelos sem orchestração master. Cross-squad handoff era ad-hoc. Agora:

1. **@ops-chief é único orchestrator** que conhece sub-chiefs (@creative-chief, @strategy-chief, @meta-ads-chief, @pt-chief)
2. **Cycle ID propagation** — UUID único atravessa todas as fases cross-squad
3. **Activity log unified** — `parent_squad='primeteam-ops'` + `sub_squad=<name>` + `cross_squad=true` em cada log entry
4. **Sub-chiefs NÃO se encadeiam direto** — sempre retornam ao ops-chief antes de próximo step
5. **Workflow YAML define sequence + handoff cards V18** — não há improvisação

## Convenção de naming

```
wf-{source-domain}-to-{target-domain}.yaml
```

Examples:
- `wf-creative-to-publish.yaml` (creative-studio → primeteam-ops)
- `wf-strategy-to-creative.yaml` (strategic-management → creative-studio)
- `wf-creative-to-ads.yaml` (creative-studio → meta-ads)

Para workflows multi-step (3+ squads): `wf-full-{flow-name}.yaml`.

## Anatomia de cross-squad workflow

Cada workflow YAML deve ter:

```yaml
name: <workflow-name>
type: cross_squad_workflow              # discriminator
master_orchestrator: "@ops-chief"
sub_squads_involved: [list]
handoff_protocol:
  cycle_id_strategy: "Generated by @ops-chief at phase 1"
  activity_log_strategy: "parent_squad + sub_squad + cycle_id + cross_squad=true"
  return_path: "Sub-chief → ops-chief sempre antes de next sub-squad"

phases:
  - id: phase_N
    agent: "@<orchestrator-or-specialist>"
    squad: "<which-squad>"
    target_squad: "<if-cross-handoff>"
    target_agent: "<if-cross-handoff>"
    inputs / actions / outputs
    quality_gate: <ID>
    veto: [conditions]

quality_gates: [list]
activity_log:
  cycle_id_propagation: "..."
  expected_log_entries: [per phase]
non_negoziabile: [rules]
```

## Trigger via @ops-chief

Cross-squad workflows são triggered pelo @ops-chief baseado em:

1. User invoca demanda em `/pto:agents:ops-chief`
2. ops-chief identifica que demanda atravessa múltiplos sub-squads
3. ops-chief seleciona workflow apropriado (consulta `workflows/cross-squad/`)
4. ops-chief executa fase 1 (triage + Cycle ID generation)
5. Workflow flui pelas 8+ fases conforme definido

User NÃO invoca cross-squad workflows diretamente — é responsabilidade do orchestrator decidir quando aplicar.

## Quality enforcement

- Cada workflow tem `non_negoziabile` block enforcing core rules (Cycle ID propagation, activity log, return path).
- Quality gates por phase (QG-OPS-* para fases ops, QG-CR-* para fases creative, etc.).
- Veto conditions per phase impedem progressão se requirements não atendidos.
